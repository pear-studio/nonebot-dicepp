# Review 文件确认与逐条回复（达成共识后再改代码）

在已有 **Review 文档**（通常由 `review-local-changes` 等流程产出）的基础上：先**确认文件与条目**，再**逐条评估并写入回复**，与用户**多轮对齐**；**仅在用户明确同意**后，才按达成一致的意见修改代码。

---

## 适用与不适用

- **适用**：用户给出或可推断出具体 review 文件路径；需要逐条表态、讨论、修订方案后再动手改仓库。
- **不适用**：尚未有书面 review、或用户要求直接改代码且未要求走确认流程（此时不要套用本技能的「先停后改」门禁）。

---

## 步骤

### 1. 确认 Review 文件

- 向用户确认（或根据上下文锁定）**唯一**的 review 文件路径；若路径模糊，列出候选并请用户指定。
- 读取全文；若文件不存在或为空，停止并说明。
- 在对话中简要复述：**文件名、主题/范围（若文中有）、条目大致数量**，避免对错文件。

### 2. 解析待处理条目

- 将文中可执行的评审点整理为**有序列表**（建议稳定编号：`R1, R2, …` 或沿用原文小节顺序）。
- 每条至少区分：**严重程度（若原文有）**、**问题陈述**、**修改建议（若原文有）**；原文含糊处先标为「待澄清」而非臆测。
- 若文档结构混乱，先在回复里给出「条目映射表」，再写入文件（见下）。

### 3. 逐条评估并写入回复（写入文件）

在 **同一 review 文件** 末尾新增一节（推荐标题：`## 逐条回复与共识状态`），或在用户指定的 **sidecar**（如 `同目录下 xxx.replies.md`）中写入，**一处为准、避免双份漂移**。

对**每一条**使用统一块结构，例如：

```markdown
### R{n} — <简短标题>

- **原文摘要**：…
- **评估**：[采纳 / 部分采纳 / 不采纳 / 待澄清] — 理由（结合代码或事实，必要时引用路径/符号）
- **拟执行改动**：具体到哪几个文件/函数、改法要点；若未达成共识写「无，待讨论」
- **共识状态**：待确认 | 已确认 | 已否决 | 已修订
```

要求：

- **逐条写全**，不可只写汇总代替逐条块。
- 评估须**对照仓库**：不确定就打开相关源码再写，禁止「可能」「或许」式敷衍；仍不确定则将该条标为 **待澄清** 并列出需要用户补充的信息。
- **不要**在本阶段直接改业务代码；仅允许修改该 review 文件（及约定的 sidecar）。

### 4. 暂停并等待用户

- 写完本轮逐条回复后，**停止改代码**。
- 请用户：**逐条或整体**确认、反驳、补充；明确说明哪些条目可以「按拟执行改动实施」，哪些要改方案或暂缓。
- 若用户提出新论点或修改意见：回到 **3**，对受影响条目**更新**对应块（保留简要修订历史可选：在块尾加一行 `*修订：…*`），直至用户对「可执行集合」表态清楚。

### 5. 达成共识后再实施

- **门禁**：仅当用户**明确指令**可以开始按 review 修改（例如「按已确认的 R1、R3 改代码」「开始实施」）后，才进入改代码阶段。
- 实施范围**不得超过**用户确认的子集；若用户未点名条目，则默认以文中 **共识状态为「已确认」** 的条目为准，缺省时再向用户要一句确认。
- 修改时对照 **拟执行改动** 与原文建议；完成后可简要告知覆盖了哪些 `Rn`，并建议用户跑测试（可配合 `run-tests` 等技能）。

### 6. 讨论循环

- 用户可随时追加讨论：重复 **3 → 4**；有新发现可新增 `R{n+1}` 或在旧条目标 **已修订**。
- **达成一致**的定义：对即将动手的条目，用户与书面「回复/拟执行改动」无未决分歧；仍存在分歧的条目不得擅自实施。

---

## 与 `review-local-changes` 的关系

- `review-local-changes`：从 git 范围生成 **新的** review 报告。
- 本技能：消费**已有** review 文件，做 **逐条共识与落地**，并控制 **何时允许改代码**。

---

## 注意事项

- **先写清、再动手**：未获明确实施授权前，禁止以「顺手修一下」扩大范围。
- 写入文件时保持 Markdown 可读；编号一旦给出，后续修订优先**原地更新同一条**，避免同一 `Rn` 多处矛盾。
- 若 review 与当前代码已不一致（分支切换、他人已合并），先指出差异，再建议更新条目或重新生成 review。
- 仍只在对用户可见的交付物中给建议；实施阶段再动代码，与 `review-local-changes`「只报告不改仓库」的阶段划分一致。
