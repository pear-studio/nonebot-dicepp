# Persona 模块 — 项目总览

> nonebot-dicepp 人格化 AI 系统
> 2026-04-08

---

## 一、项目定位

在 TRPG 骰子机器人 nonebot-dicepp 中集成一个 **人格化 AI 角色**。这不是一个通用聊天机器人，而是一个有性格、有记忆、有情感、会主动找你聊天的虚拟角色。

核心体验：**像和一个真人交朋友** — 从陌生到熟悉，她会记住你说过的话，会分享自己的生活，关系好了说话方式也会变。

---

## 二、核心能力

### 2.1 深度人格

通过 YAML 角色卡定义角色的一切 — 性格、背景、说话风格、示例对话。角色卡格式兼容 SillyTavern V2 标准，可复用社区生态。

### 2.2 四层记忆

| 层 | 作用 | 说明 |
|---|---|---|
| 短期记忆 | 记住刚才聊了什么 | 字数限制（如3000字），保留最近对话 |
| 用户档案 | 记住你是谁 | 从对话中提取的结构化事实（名字、爱好、宠物...） |
| 群聊观察 | 知道群里发生了什么 | 被动旁听群消息，动态触发，LLM按需查询 |
| 角色日记 | 角色自己的生活 | 全天实时模拟生活事件，晚上总结为日记 |

**注**：去掉对话摘要，简化系统；用户档案提前到Phase 1实现，是核心情感价值所在。

### 2.3 好感度系统

四维隐藏指标（亲密度/激情/信任/安全感），用户只看到**区间标签**，标签文案由角色卡定义：

```yaml
# 角色卡 extensions.persona 字段，6 个标签依次对应 0-10/10-20/20-40/40-60/60-80/80-100
warmth_labels: ["漠然", "疏冷", "普通", "友好", "亲近", "心意相通"]
```

- 初始值由角色卡定义（`initial_relationship` 字段），默认 30（第三区间）
- 好感度跌至"厌倦"区间（0-10）时，角色有概率拒绝回复（普通指令仍响应）
- 长时间不互动会缓慢衰减，但不会低于初始值（衰减下限 = 角色卡初始值 + 20）

好感度影响：说话语气、分享深度、主动程度、回复速度等。

### 2.4 角色生活模拟

角色有自己的"生活" — 不是一次性编造，而是全天实时模拟：

- 白天：System Agent 按计划在一天内生成若干生活事件，Character Agent 产生内心反应  
- **事件时刻从哪里来**：由**角色卡** `extensions.persona` 配置 — `daily_events_count`（一天几条）、`event_day_start_hour` / `event_day_end_hour`（活跃窗口）、`event_jitter_minutes`（每个时间槽的随机抖动）。代码侧用 `PersonaExtensions.generate_event_times()` 在窗口内均分再抖动，得到具体触发时刻；`persona_ai` **没有** `character_life_event_hours` 之类字段。  
- 晚上：Character Agent 把全天事件总结成一篇日记（`persona_ai.character_life_diary_time` 等；与事件分布无关）  
- 所有用户看到的是同一个角色的同一天，但根据关系亲疏，分享的内容深度不同

### 2.5 主动消息

角色会主动找你聊天，基于日记事件驱动：

| 类型 | 触发 | 说明 |
|---|---|---|
| 事件分享 | 角色生活事件生成后 | 按好感度优先级发送，私聊个性化，群聊通用 |
| 探索配额 | 事件分享时的低好感度扩展 | 对近 7 天有互动的活跃用户（好感度 20-40），保留少量分享机会，防止关系冷启动 |
| 想念触发 | ≥3天未互动 | 从最近日记事件中取一条作为"由头"，私聊 only |
| 一定触发事件 | 指定时间段内必生成 | 问候/作息等，角色卡配置 time_range |

**发送策略**：
- 私聊：按好感度从高到低，结合用户档案个性化
- 群聊：群活跃度≥60才发送，最多10条/5分钟
- 从好感度高用户开始，最后发送群聊
- 安静时段（23:00-07:00，根据角色作息调整）和冷却间隔

**群聊活跃度**：
```
活跃度 = min(100, max(0, 当前分数 - 时间衰减 + 互动加分))
时间衰减（分层策略）：
- 24小时内有互动 → 不衰减
- 24小时内有内容 → 衰减减半（-5/天）
- 完全无内容 → 正常衰减（-10/天）
互动加分：每次@/聊天 +2分（日上限20）
```
> 注：**群白名单内的群**活跃度不会低于初始值（50），只关闭主动推送，不影响被动 @ 回复。

### 2.6 成本控制

- 主模型（贵）：只用于对话回复，每人每天限额 20 次
- 辅助模型（便宜）：用于评分、摘要、事件生成等后台任务，不限额
- 用户自带 API Key：可独立配置主模型和辅助模型（key/url/model），解除限额
- **白名单用户免限额**：白名单功能激活时，白名单内的用户不受每日限额约束（主动选择加入，视为受信任）；并发限制（Semaphore）仍保留
- Prompt caching：角色卡 system_prompt 是静态内容，放在 messages 最前面，OpenAI/Anthropic 等主流 API 会自动识别并缓存，无需额外代码即可节省 ~50% system token 计费

### 2.7 工具调用（渐进引入）

角色可以使用工具：搜索深层记忆（解决"失忆"问题）、掷骰子（TRPG 场景），后续可扩展网络搜索等。Phase 1 不使用工具，纯上下文注入。

## 三、场景模式

### 访问控制：默认关闭

**AI 对话功能默认对所有用户/群聊关闭。** 本模块的主体是 TRPG 骰子机器人，AI 伴侣是可选的附加体验。

访问控制通过白名单实现，白名单分两类：
- **用户白名单**：用户私聊使用口令自助加入，加入后可在私聊中使用 AI 对话
- **群聊白名单**：管理员直接将群加入，加入后该群的 @ 消息会被 AI 回复

白名单功能的开关：**管理员设置口令后白名单才生效**。未设置口令时白名单不激活（不清除已有数据），所有用户均可访问 AI（视为开发/测试状态）。

未在白名单内的用户/群发送 `.ai` 命令或 @ 机器人聊天，bot 静默忽略，不干扰正常 TRPG 流程。

```
┌──────────────────────┬────────────────────────────┐
│       私聊模式        │         群聊模式            │
├──────────────────────┼────────────────────────────┤
│ 完整人格 + 好感度     │ 聊天 + 简化好感度           │
│ 好感度 per-user      │ 好感度 per-user-in-group    │
│ 衰减 ✅ 事件 ✅      │ 衰减 ❌ 事件 ❌             │
│ 完整记忆             │ 简化记忆（无日记/事件注入）   │
│ 触发: @bot           │ 触发: @bot（仅 @ 消息）     │
│ 默认关闭，用口令加入  │ 默认关闭，管理员加入群白名单  │
│                      │ 被动旁听群消息并记忆         │
└──────────────────────┴────────────────────────────┘
```

> 注：群聊被动旁听（不被 @ 时静默记录消息）不受白名单限制，仅 AI 对话回复需要白名单。

### 群聊被动观察机制

角色静默旁听群消息，动态触发提取：

**触发条件**（实时自适应）：
- 初始阈值：20条 或 2小时
- 根据填充速度动态调整：
  - <30分钟填满 → +10条（最多60条）
  - >3小时填满 → -5条（最少5条）

**过滤规则**：丢弃图片/表情包/语音、纯emoji、太短(<5字)、太长(>500字)、指令消息

**使用方式**：
- 提供查询接口，LLM按需检索（非全量注入，**Phase 2g 实现**）
- 每群最多保留30条观察记录
- 私聊只引用与当前用户相关的观察

### 群聊记忆粒度

| 维度 | 粒度 | 说明 |
|---|---|---|
| 聊天好感度 | per-user-in-group | 每人在每个群有独立的关系状态，影响对话温度 |
| 群聊活跃度（未来） | per-group | 群整体活跃度，影响主动发消息等功能 |
| 对话历史/摘要 | per-group | 群聊语境共享 |
| 用户档案 | per-user 跨群 | 跨群有效（"小明养了柯基"） |
| 观察记录 | per-group + participants | 记录参与者，私聊中可按 user_id 检索 |

> **关系状态双层模型说明**：  
> - 聊天好感度（per-user-in-group）：记录每个用户与 AI 之间的关系温度，决定 AI 对该用户的回复风格。每人行为只影响自己的好感度。  
> - 群活跃度（per-group，Phase 4+ 实现）：记录整个群的互动活跃程度，用于控制 AI 是否主动发言、发送频率等。两者相互独立。

## 四、系统架构

```
┌─────────────────────────────────────────────────────────────┐
│                    QQ / OneBot V11                           │
└────────────────────────┬────────────────────────────────────┘
                         │
              ┌──────────▼──────────┐
              │   PersonaCommand    │  @bot 触发聊天 / .ai 触发命令
              │   + tick() 定时钩子  │  主动消息 / 生活模拟
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │  PersonaOrchestrator│  编排层：调度各 Agent
              └──┬─────┬─────┬──┬──┘
                 │     │     │  │
          ┌──────▼┐ ┌──▼──┐ ┌▼──▼───────┐
          │ 主人格 │ │评分 │ │ 生活模拟    │
          │ Agent │ │Agent│ │System+Char │
          │(主模型)│ │(辅助)│ │  (辅助)    │
          └───────┘ └─────┘ └────────────┘
                 │     │     │
          ┌──────▼─────▼─────▼──────────┐
          │    四层记忆 + 关系状态        │
          │       SQLite 持久化          │
          └─────────────────────────────┘
```

### 多 Agent 职责

| Agent | 模型 | 职责 | 实现 |
|---|---|---|---|
| 主人格 Agent | 主模型（贵） | 生成对话回复 | `agents/persona_agent.py` |
| 评分 Agent | 辅助模型（便宜） | 批量分析对话 → 好感度变化 | `agents/scoring_agent.py` |
| System Agent | 辅助模型 | 生成角色生活事件 | `agents/event_agent.py` |
| Character Agent | 辅助模型 | 对事件产生反应 + 总结日记 | `agents/event_agent.py`（同文件） |
| 观察 Agent | 辅助模型 | 从群聊中提取值得记忆的内容 | orchestrator 内联调用 LLM |
| 档案提取 Agent | 辅助模型 | 从对话中提取用户事实 | 与评分 Agent 合并为一次调用 |

> 注：只有前 3 行是独立 Agent 文件。观察/摘要/档案提取本质是"换一个 prompt 调用 LLM"，不需要独立 Agent 类。

## 六、用户命令

### 普通用户

| 命令 | 说明 |
|---|---|
| `.ai join <口令>` | 用口令将自己加入白名单，开启 AI 对话（**仅限私聊**） |
| `@bot <消息>` | 与角色对话（需在白名单内） |
| `.ai` | 查看角色自我介绍 |
| `.ai status` | 查看 AI 状态及白名单情况 |
| `.ai profile` | 查看好感度区间/趋势、认识天数、互动次数、角色记住的用户信息 |
| `.ai mute` | 关闭主动消息（**Phase 3 实现**） |
| `.ai unmute` | 开启主动消息（**Phase 3 实现**） |
| `.ai clear` | 清空对话历史（调试用，不清除好感度） |
| `.ai key` | 查看/配置自己的 API Key（**Phase 4 实现**） |
| `.ai key config ...` | 表单式配置（**Phase 4 实现**） |
| `.ai key clear` | 清除自定义 API Key 配置（**Phase 4 实现**） |

### 管理员命令

| 命令 | 说明 |
|---|---|
| `.ai admin code <新口令>` | 设置/更新口令（设置后白名单功能激活；不踢出已有成员） |
| `.ai admin code clear` | 清除口令（白名单功能停用，已有成员数据保留） |
| `.ai admin whitelist` | 查看白名单列表（用户 + 群聊，含加入时间） |
| `.ai admin whitelist add group <group_id>` | 将指定群加入白名单 |
| `.ai admin whitelist remove <user_id>` | 移除指定用户 |
| `.ai admin whitelist remove group <group_id>` | 移除指定群 |
| `.ai admin whitelist clear` | 清空整个白名单（需二次确认） |
| `.ai admin trace <user_id>` | 导出该用户最近 5 次 LLM 调用 trace（Phase 7a） |
| `.ai admin stats` | 查看今日 LLM 调用统计（延迟、错误率、token 消耗）（Phase 7a） |
| `.ai admin errors` | 查看最近 24h LLM 错误摘要（Phase 7a） |

> 管理员身份通过 DicePP 已有的 `is_admin(user_id)` 鉴权，无需额外实现。

## 七、实施路线图

```
Phase 1: 骨架与基础对话 + 用户档案（约 4 天）
  1a 模块骨架 — 目录结构 + 空命令注册 + 配置模型
  1b 角色系统 — YAML 角色卡加载
  1c LLM 路由 — 多模型客户端
  1d 基础对话 — 角色化单轮/多轮对话
  1e 持久化   — 对话历史 SQLite 存储
  1f 白名单   — 暗号入场 + 管理员命令
  1g 用户档案 — 从对话提取结构化信息（从原Phase 4提前）

Phase 2: 主动消息 + 群聊观察 + 时间衰减（约 6 天，已完成）
  2a 时间衰减 — 长时间不互动好感度下降
  2b 一定触发事件 — 问候/作息等定时事件
  2c 好感度驱动分享 — 根据好感度决定分享优先级
  2d 角色生活模拟 — 全天实时模拟+日记总结
  2e 事件分享 — 私聊+群聊策略（群活跃度、限额）
  2f 想念触发 — ≥3天未互动时的主动消息
  2g 群聊观察 — 被动旁听群消息并提取信息

Phase 3: 上下文升级 + Function Calling（约 3 天）
  3.0 主动消息开关 — `.ai mute/unmute` 用户控制
  3a 四层记忆集成 — 日记注入上下文，缩短短期记忆（3000→1500字）
  3b Function Calling 基础设施 — 多轮工具调用（最多5轮）
  3c search_memory 工具 — 搜索用户档案/观察/日记
  3d 厌倦拒绝机制 — 好感度0-10时概率拒绝回复（补充Phase 2）

Phase 4: 成本与配置                                  约 2 天
  4a 配额系统 — 每日用量追踪 + 超额拒绝
  4b 用户自带 Key — 表单配置 + 加密存储
  4c 掷骰工具 — roll_dice + DicePP 引擎复用

Phase 6: 深度人格（可选，约 5 天）
  6a 世界书引擎 — 关键词触发知识注入（优先）
  6b Drive 系统 — 5 维驱动 + 时间代谢（可选）

Phase 7: 发布与可观测性（约 5 天）
  7a 调试数据与 LLM Trace — prompt/response/tool_calls 持久化与调试导出
  7b 敏感词过滤 — sensitive-word-filter
  7c 文档与测试
```

每个 Phase 结束都是一个**可独立使用**的产品：
- Phase 1 完成 = 有性格、有记忆的 AI 聊天
- Phase 2 完成 = + 好感度系统 + 主动消息 + 生活模拟
- Phase 3 完成 = + 深层记忆搜索（不再"失忆"）
- Phase 4 完成 = + 成本可控 + 用户可自配模型

## 八、关键设计决策

| 决策 | 结论 | 理由 |
|---|---|---|
| 框架 | 自研轻量编排 | 对话场景不需要 ReAct/tool-calling 重框架 |
| 数据库 | SQLite | DicePP 已有 Repository 模式 |
| 角色卡 | YAML only | 手写友好，后续可扩展 |
| 好感度 | 纯渐变，无章节/Boss | 更自然，更简单 |
| 好感度初始值 | 角色卡定义，默认 30 | "从陌生到熟悉"，初始落在"疏远"区间 |
| 主动消息 | 私聊+群聊均可 | 可按 schedule 配置 scope |
| 降级 | 返回错误提示 | 不回退到 core/persona 正则 |
| 角色日记 | 实时模拟→总结 | 比一次性编造更自然 |
| 群聊观察 | participants 关联 | 私聊中可引用群里和用户相关的事 |
| 评分频率 | 每 5 轮批量 | 省 80% API 成本 |
| 用户 Key | 可配 primary+auxiliary | 自由度等同开发者 |
| LLM JSON 解析 | 3 级容错 + Pydantic 校验 + zero-delta 兜底 | 直接解析→去 markdown→正则提取，失败跳过评分 |
| 并发控制 | asyncio.Semaphore(N) | 便宜方案通常只允许 1-2 并发，排队比报错好 |
| Token 预算 | Phase 1 只限轮数；Phase 4c 加 TokenBudgetManager | 渐进式，Phase 1 不需要 tiktoken |
| .ai clear | 保留为调试工具 | 只清对话历史，不清关系状态；好感度是"关系"不是"记忆" |
| LLM 重试策略 | 指数退避（2→4→8 秒），最多 3 次 | 提高 API 调用成功率，减少用户看到错误提示 |
| 想念触发 | ≥3天未互动，从近期日记随机取由头 | 有"由头"更自然；无需追踪"未分享事件"（想念只触发一次，必然有未发事件） |
| 记忆层数 | 四层，去掉摘要 | 简化系统，用户感知不明显 |
| 用户档案 | Phase 1实现 | "记得我说过的话"是核心爽点 |
| 群聊观察 | 动态触发 + LLM查询 | 适应跑团波动性，避免API浪费 |
| 永久状态 | 文本格式，LLM工具读写 | 灵活记录角色长期变化 |
| 探索配额 | 事件分享时保留少量给活跃低好感用户（好感度20-40，近7天有互动） | 防止新用户冷启动 |
| 角色区间标签 | 角色卡 `warmth_labels` 字段，6个标签列表 | 贴合角色人设 |
| 群白名单活跃度 | 白名单群活跃度下限=初始值(50)，只关闭主动推送 | 被动 @ 回复不受活跃度影响 |
| .ai mute | 用户开关主动消息 | 防骚扰，开关比时长设置更直觉 |
| .ai profile | 合并 .ai status，展示好感度+趋势+用户档案 | 一个命令看清所有关系状态 |
| 默认访问控制 | 白名单默认空，口令入场 | 主体是 TRPG，AI 伴侣是可选体验，不干扰其他用户 |
| 白名单类型 | 用户白名单（口令自助）+ 群聊白名单（管理员直接添加） | 两种场景访问路径不同 |
| 白名单作用域 | 私聊=用户白名单；群聊=群白名单（互相独立） | 群白名单不依赖用户白名单，管理更清晰 |
| 口令提交方式 | 仅私聊可用 `.ai join` | 防止口令在群内被他人看到后批量加入 |
| 白名单激活条件 | 口令已设置才生效；清除口令则停用但保留数据 | 口令是开关，不清数据避免误操作 |
| 白名单用户免限额 | 白名单激活时，白名单内用户/群跳过每日配额 | 主动加入的用户视为受信任；Semaphore 保留 |
| 口令更新行为 | 不踢出已有成员 | 口令只是入场凭证，不是持续身份验证 |

## 十、风险

| 风险 | 应对 |
|---|---|
| LLM 延迟/超时 | timeout 配置 + 错误提示 |
| LLM 返回格式异常 | 3 级 JSON 容错（直接解析→去 markdown→正则提取）+ zero-delta 兜底 |
| 并发请求被模型拒绝 | asyncio.Semaphore 限流，超并发自动排队 |
| 角色"出戏" | 强 system_prompt + 示例对话 + warmth_level 行为指引 |
| 恶意用户骚扰 | 极端负评分触发 30 分钟冷却，冷却期拒绝回复且不消耗配额 |
| API 成本失控 | 只限主模型次数 + 用户自带 Key |
| 记忆"失忆" | Phase 3 已引入 search_memory 工具主动检索 |
| 群聊消息量大 | 观察缓冲区批量处理，不逐条分析 |
| 口令泄露 | 强制私聊提交，bot 回复不含口令明文；管理员可随时更换口令 |
| 白名单滥用 | 管理员可查看列表并移除问题用户/群 |
