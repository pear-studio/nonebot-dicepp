## Context

### 背景

现有的 dice_hub 模块通过 QQ 消息在机器人之间进行通信，存在以下问题：
1. 依赖 QQ 账号在线状态
2. 受 QQ 消息频率限制
3. 账号可能被封禁，存在风险

### 当前状态

- 现有模块：`module/dice_hub/` 包含 hub_command.py, manager.py, data.py, encrypt.py
- 通信方式：通过 QQ 私聊消息交换名片和同步信息
- 数据存储：本地 JSON 文件存储机器人好友信息

### 目标

通过网站 API 实现机器人注册与发现，替代现有的 QQ 消息通信方式。

---

## Goals / Non-Goals

**Goals:**
- 机器人可通过 HTTP API 向网站注册，获取 API Key
- 机器人定期向网站发送心跳，维持在线状态
- 管理员可通过指令查看在线机器人列表
- 管理员可查看/管理机器人 API Key
- 保留 `.hub` 指令触发方式，用户迁移成本低

**Non-Goals:**
- 跨机器人掷骰功能（暂不考虑）
- 角色数据共享（暂不考虑）
- 网站端实现（由另一个项目负责）
- 数据存储功能（角色卡等暂不考虑）

---

## Decisions

### 1. 通信方式：HTTP API vs WebSocket

**选择：HTTP API**

| 特性 | HTTP API | WebSocket |
|------|----------|-----------|
| 实现复杂度 | 低 | 高 |
| 资源消耗 | 低 | 较高 |
| 实时性 | 心跳间隔决定 | 实时 |
| 网站兼容性 | 简单 | 需要额外支持 |

**理由**：网站项目已是 Django REST Framework，HTTP API 实现简单且兼容性好。心跳间隔 10 秒（测试）/ 3 分钟（正式）可满足需求。

### 2. HTTP 库选择：aiohttp vs httpx

**选择：httpx**

| 特性 | aiohttp | httpx |
|------|---------|-------|
| 异步支持 | 原生 | 原生 |
| NoneBot2 兼容性 | 需要适配 | 良好 |
| 同步调用 | 不支持 | 支持 |

**理由**：httpx 同时支持同步/异步，DicePP 部分代码仍使用同步方式，httpx 更灵活。

### 3. API Key 格式

**选择：32 位随机字符串（URL-safe Base64）**

- 长度：32 字符
- 格式：字母 + 数字，无特殊字符
- 生成：由网站端负责

### 4. 心跳机制

| 阶段 | 心跳间隔 | 离线判定 |
|------|----------|----------|
| 测试 | 10 秒 | 30 秒无心跳 |
| 正式 | 3 分钟 | 10 分钟无心跳 |

**实现**：使用 NoneBot2 的 `scheduler` 插件或定时任务

### 5. 配置项设计

```python
# 新增配置项
CFG_HUB_API_URL = "dicehub_api_url"  # 网站 API 地址

# 已有配置项（复用）
CFG_HUB_NAME = "dicehub_name"  # 机器人昵称
CFG_MASTER = "master"  # Master 账号
```

### 6. 指令设计

| 指令 | 说明 | 权限 |
|------|------|------|
| `.hub register` | 向网站注册机器人 | Master |
| `.hub key` | 查看当前 API Key | Master |
| `.hub list` | 查看在线机器人列表 | Master |
| `.hub online` | 手动触发心跳 | Master |

## Risks / Trade-offs

### 1. 网站服务不可用

**风险**：网站服务宕机时，机器人无法注册/心跳

**缓解**：
- 心跳失败不影响机器人本地功能
- 离线机器人保留本地缓存
- 网站恢复后自动重连

### 2. API Key 泄露

**风险**：API Key 被恶意获取

**缓解**：
- API Key 仅存储在机器人本地配置
- 传输使用 HTTPS
- 网站端可绑定 IP 白名单

### 3. 旧数据迁移

**风险**：删除旧模块导致数据丢失

**缓解**：
- 旧数据存储在 `dicehub` 数据块中
- 删除前提示用户备份
- 新的在线列表由网站提供，本地无需存储

### 4. 网络延迟

**风险**：心跳请求超时

**缓解**：
- 设置合理超时（如 10 秒）
- 心跳失败不立即标记离线
- 重试机制

## Migration Plan

### 步骤 1：新增配置项
- 在 `core/config/common.py` 新增 `CFG_HUB_API_URL`

### 步骤 2：重构 dice_hub 模块
- 删除：encrypt.py, data.py（数据模型由网站管理）
- 重写：hub_command.py, manager.py
- 新增：api_client.py（HTTP 请求）

### 步骤 3：测试
- 本地测试注册、心跳、列表功能
- 与网站端联调

### 步骤 4：上线
- 部署新版本机器人
- 通知用户重新注册

### 回滚
- 如有问题，可回滚到旧版本
- 旧数据保留在数据文件中
