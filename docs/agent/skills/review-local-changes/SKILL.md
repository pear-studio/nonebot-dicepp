# 本地改动与分支提交 Review

对当前工作区进行客观评审：

1. **未提交的改动**（与 `HEAD` 的差异）
2. **若在非主干分支**：相对**主干**（默认分支）**多出来的全部提交**所引入的改动（合并基之后的提交历史及其代码）

评审维度：实现质量、设计平衡（避免过度设计与短视决策）、长远影响；对发现的问题须给出**可执行的修改建议**（说明改什么、怎么改、改动的边界或风险），而非只下结论。

---

## 步骤

### 1. 获取评审范围

#### 1a. 本地未提交改动

```powershell
git diff HEAD          # 全部本地改动（含已暂存 + 未暂存）
git diff --stat HEAD   # 改动文件概览
```

#### 1b. 主干与「领先于主干的提交」（仅非主干时需要）

解析**主干**引用（按顺序尝试，取第一个存在的）：

```powershell
# 常见：origin 的默认分支（如 origin/main）
git symbolic-ref --short refs/remotes/origin/HEAD 2>$null
# 若无，再试 origin/main、origin/master、main、master 等
```

当前分支：

```powershell
git rev-parse --abbrev-ref HEAD
```

若当前分支为 `HEAD`（detached）或无法可靠判断，向用户说明并仍以 **1a** 的本地 diff 为主；若能确定主干引用且**当前分支不是主干**（或与主干 `HEAD` 提交不同），则补充收集相对主干的提交与补丁：

```powershell
# 领先主干的提交列表（主干..当前 HEAD）
git log <trunk-ref>..HEAD --oneline

# 相对合并基到当前 HEAD 的累积 diff（推荐，与 merge 行为一致）
git diff <trunk-ref>...HEAD
git diff --stat <trunk-ref>...HEAD
```

说明：`<trunk-ref>` 替换为实际主干远程分支，如 `origin/main`。三点 diff `A...B` 表示从 `merge-base(A,B)` 到 `B` 的改动。

**退出条件**

- 若 **1a 无改动** 且 **1b 无领先提交**（或不在非主干场景），输出「暂无本地修改，且当前分支相对主干无额外提交」并退出。
- 若仅有其一有内容，则只把有内容的部分交给后续 Agent（报告中对应小节可写「无」）。

### 2. 启动两个并行子 Agent

启动两个子 Agent，**同时启动，不等待其中一个完成再启动另一个**。
让他们自己去看修改，不需要把修改直接传递过去。
如果启动失败，不要一个人继续，而是向用户确认情况。

每个 Agent 的 prompt 应包含：

1. **完整 diff 内容**：至少包含 `git diff HEAD`；若在非主干且存在领先提交，还须包含 `git diff <trunk>...HEAD`（可与本地 diff 分块标注，避免混淆）。
2. 可选：`git log <trunk>..HEAD --oneline` 作为提交脉络上下文。
3. 明确的聚焦方向（见下）。
4. 输出要求：
   - 列出发现的问题（按严重程度：严重/警告/建议）。
   - **每一条**问题（严重、警告优先；纯优化类「建议」若可落地也写）须附带 **修改建议**：具体步骤或伪代码级说明、涉及文件/函数若可指出则写出；若存在多种改法，可简述取舍。
   - 若某条仅为正面观察、无改进点，不必硬凑建议。
   - 无问题时也要明确说「此方面未发现问题」。

**Agent A — 实现质量**，聚焦：

- 正确性：逻辑是否有 bug，边界条件是否处理
- 健壮性：错误处理、空值、异常分支
- 测试：改动是否有对应测试，或现有测试是否需要更新
- 代码风格：是否符合项目已有规范（命名、注释、结构）

**Agent B — 设计质量**，聚焦：

- 过度设计识别：不必要的抽象层、提前泛化、「以防万一」的冗余代码
- 短视识别：硬编码常量、缺少迁移路径的 breaking change、未来明显会痛的耦合
- 一致性：是否与项目已有模式保持一致，还是引入了孤立的新范式
- 长远可维护性：改动是否在清偿技术债，还是在累积新的技术债

评审时要求子 Agent **同时审视**「工作区未提交部分」与「相对主干的提交累积」（若存在），并在结论中区分或合并说明，避免只看了其中一半。

### 3. 等待两个 Agent 返回结果

收集两个 Agent 的完整输出，但不要盲信，需要再次评估和确认正确性；整理报告时**保留**各条问题附带的修改建议，勿压缩成仅含标题的清单。
等待期间可使用 DicePP Shell 等工具验证改动实际效果，将验证结果纳入最终评估。

### 4. 综合输出报告

将报告写入项目根目录的 `.temp/` 中的 `.md` 文件，文件名格式为 `review-YYMMDD-HHMM-主题`。

格式如下：

```
## 本地改动与分支 Review

**评审范围**
- 主干引用：<trunk-ref 或「未使用」>
- 当前分支：<branch>
- 本地未提交：<有/无>（文件数、+/- 行数若有则写）
- 相对主干的额外提交：<有/无>（提交数、简要列表若有则写）

---

### 实现质量
[Agent A 结论：按严重/警告/建议分级；每条问题采用「问题描述 → 修改建议」结构，无问题时写「未发现问题」]

---

### 设计质量
[Agent B 结论：同上]

---

### 修改建议汇总（可选但推荐）
[将上述两节中的修改建议按优先级合并为简短清单：先做哪几条、可延后哪几条，便于落地；若无建议则写「无」]

---

### 综合评估
[1-3 句话总结：改动整体质量如何，最需要关注的点是什么，是否建议直接提交 / 是否建议先处理分支内历史提交再合并]
```

---

## 注意事项

- 需要逐行阅读所有修改（含相对主干的累积 diff），不可望文生义，不可草率下结论。
- 不确定的直接进一步看源码分析，不要在汇报中出现「若有时间可进一步确认」「若项目已有」这样的话。
- 保持客观：不因改动量小就降低标准，也不因改动量大就放宽尺度。
- 严重问题（正确性 bug、数据安全）必须明确标出，不可淡化。
- 对「过度设计」与「不够健壮」的判断要结合项目当前阶段，不套用通用最佳实践教条，但新的设计或重构设计时应当参考业界通用做法。
- 只报告、不自动改仓库；**修改建议**以文档形式写入报告，须具体可执行，避免空泛的「建议加强测试」「建议重构」式表述。
